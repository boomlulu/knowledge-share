# H5 分镜 · 审核总览

> 用途：低成本审核演讲逻辑，不看画面细节。  
> 看法：只检查三件事：这一镜头有没有必要、还债是否清楚、新钩子是否自然。

---

## 总追问链

画面看起来同步，为什么工具还会报警？  
→ 工具看见的到底是什么？  
→ 小到肉眼看不见的差异，为什么会变成整个世界分叉？  
→ 既然帧同步这么难，能不能让服务器来算？  
→ 状态同步也很贵，那有没有第三种复杂度分配方式？  
→ 客户端有权威，那 PvP 怎么办？  
→ 一个成熟同步框架，到底应该管理什么？

---

## 一行审核表

| 镜头 | 核心作用 | 还债 | 新钩子 |
|------|----------|------|--------|
| 01 肉眼看完全同步 | 让观众先相信画面没问题 | 无，建立开场事实 | 画面同步，为什么我还不放心？ |
| 02 透明叠加 | 用半透明叠图打破“完全同步”的直觉 | 肉眼只能看见表现，不一定能看见底层数据差异 | 这点差异是真问题，还是显示误差？ |
| 03 工具刷日志 | 用检测工具制造第一波冲击 | 证明这不是主观怀疑，工具确实报警 | 画面还没穿帮，工具凭什么报警？ |
| 04 画面数据分离 | 建立“画面 vs 底层数据”的核心区分 | 工具看的不是像不像，而是底层数据是否一致 | 底层数据差异从哪来？ |
| 05 动画驱动暗门 | 点出真实根因线索 | 差异来源之一是动画驱动逻辑没有被定点数约束 | 这么小的漏口，为什么会变成世界报警？ |
| 06 工具不是修复 | 把工具价值拔高为企业级门槛 | 看见问题比马上修问题更靠前 | 看见不同步后，下一步最难的是什么？ |
| 07 雪崩前一帧 | 从工具报警转向根因追溯 | 工具只是开始，真正难的是找到最早那一帧 | 一帧小差异凭什么滚成整个世界对不上？ |
| 08 MOBA 场景 | 切入第二个真实踩坑案例 | 无，建立案例舞台 | 普通 MOBA 场景里，分叉最早发生在哪？ |
| 09 帧同步隐藏条件 | 解释为什么底层数据差异不能忽略 | 帧同步要求每一帧底层状态一致 | 哪个普通逻辑会成为分叉起点？ |
| 10 小兵选塔 | 把确定性问题落到具体判断 | 无，制造分叉前提 | 如果两个客户端算出不同结果，会怎样？ |
| 11 分叉开始 | 展示世界分叉第一瞬间 | 最早分叉可能只是小兵选左塔还是右塔 | 只是小兵走向不同，为什么影响整个世界？ |
| 12 雪崩链条 | 完成“小差异变大问题”的核心解释 | 后续每一帧都基于已分叉状态继续计算 | 客户端算一致这么难，能不能别让客户端算？ |
| 13 帧同步工程压力 | 把案例升级成工程判断 | 帧同步要求整个模拟层保持确定性 | 服务器能不能接管这个压力？ |
| 14 让服务器算 | 让状态同步作为自然逃路登场 | 可以让服务器决定结果，客户端接收状态 | 这招这么自然，为什么不直接用？ |
| 15 状态同步价值 | 先承认状态同步确实有用 | 状态同步能绕开一部分客户端确定性压力 | 它把压力转移到哪里？ |
| 16 服务器变重 | 揭示状态同步工程代价 | 服务器要理解并运行足够完整的游戏世界 | 两条路的本质区别是什么？ |
| 17 复杂度换位置 | 把两种同步升成架构取舍 | 帧同步和状态同步是把复杂度放在不同位置 | 有没有第三种复杂度分配方式？ |
| 18 第三种方式 | 让自研框架作为困境答案登场 | 用本地拟真驱动 + 分散权威修正重新分配复杂度 | 这套框架一句话怎么说？ |
| 19 一句话框架 | 给出框架总定义 | 拟真驱动每一帧，权威状态拉回偏差 | 拟真驱动意味着什么？ |
| 20 拟真驱动 | 解释本地仍然推进世界 | 客户端按同一套规则推进，但不承诺每帧完全一致 | 既然不绝对一致，偏了怎么办？ |
| 21 权威修正 | 解释偏差如何被控制 | 用权威状态周期性把本地状态拉回可接受范围 | 权威从哪里来？ |
| 22 分布式权威 | 澄清分布式不是很多服务器 | 权威不是来自服务器集群，而是分散在客户端和对象上 | 玩家和非玩家对象分别谁负责？ |
| 23 持有者 | 解释权威如何落到对象上 | 玩家对自己负责，实体由对应持有者负责 | 持有者固定吗？ |
| 24 权威转移 | 解释权威动态性 | 对象权威可从一个持有者转移到另一个持有者 | 框架追求永不出错，还是出错后可控？ |
| 25 管理错误 | 提炼框架哲学 | 不追求永不出错，而是追求偏差可观测、可修正、可收敛 | 玩家自己有权威，会不会作弊？ |
| 26 防作弊反问 | 主动制造最大质疑 | 不还债，提出 PvP 公平性问题 | 客户端权威是不是等于相信客户端？ |
| 27 可回放审计 | 回应客户端权威信任问题 | 不是无条件信任，而是让行为留下可回放证据 | 回放成本会不会太高？ |
| 28 分级成本 | 回应回放审计成本 | 异步抽样、高风险加频、严重时实时模拟 | 这套框架真正管理的是什么？ |
| 29 四个判断问题 | 把全场技术点收成判断框架 | 管理的是看见偏差、处理偏差、分配复杂度、管理信任 | 能不能用一句话收住？ |
| 30 回到开场画面 | 用开场画面做最终呼应 | 检测、修正、回放三条能力一起回应开场问题 | 真正要回答的问题是什么？ |
| 31 最后一句 | 完成大兑现 | 偏了以后能否看见、拉回、解释，完成全场闭环 | 无 |

## 快速审核清单

- 是否有某个镜头没有承担“还债”或“抛钩子”的作用？
- 是否有某个钩子下一镜头没有接住？
- 是否有某个真实案例被讲成了泛泛概念？
- 是否有某段从“架构取舍”滑向“技术教程”？
- 是否有某个术语会和后文冲突，比如“状态同步”？
- 是否有某个镜头可以合并或删除而不影响追问链？
